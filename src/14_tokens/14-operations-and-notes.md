# 运维、版本与「CoinLock」说明

## 导读

代币上线后，**权限对象**（**`TreasuryCap`、`MetadataCap`、`DenyCapV2`、`TokenPolicyCap`**）的保管方式决定事故面。本节列出 **常见做法** 与 **版本查阅** 建议；并说明当前 Framework 中与 **锁定** 相关的常见误解。

---

## 权限分割

| 对象 | 风险 | 常见做法 |
|------|------|----------|
| **`TreasuryCap`** | 无限铸 / 销毁供应 | 多签、时间锁、或拆成 **热钱包 + 冷钱包** 分阶段铸币 |
| **`DenyCapV2`** | 黑名单写入 | 合规团队专用；与 **技术运维** 分离 |
| **`MetadataCap`** | 展示信息更新 | 市场或运营；变更需公告 |

## DenyList 的 epoch 语义

**`deny_list`** 对配置的修改往往 **在下一 epoch 生效**（见模块内 **Config / next_epoch** 设计）。运营侧 UI 应提示 **「已提交 / 未生效」**，避免用户误以为即时封禁或即时解封。

## 升级与迁移

包 **升级**（见本书第十八章）可能涉及 **`Currency`** 状态迁移、**`DenyCap` → `DenyCapV2`** 等路径；**务必**以部署目标网络的 **Release Note** 与 **Framework 源码** 为准，本书示例不替代审计。

## 「CoinLock」与质押

在当前仓库随 **Sui 1.69** 构建的 **`sui-framework`** 依赖中 **未提供** 名为 **`CoinLock`** 的通用类型（若你见到该词，可能来自 **旧文档、其他链**，或 **质押 / 锁仓** 业务合约的俗称）。若需要 **时间锁或线性释放**，应在 **应用层** 用 **共享对象 + `Balance` + 时钟 `Clock`** 自行实现，并对 **`TreasuryCap`** 的接触面做审计。

## 小结

**安全发币 = 类型设计（Coin/Token） + 权限保管 + 合规开关 + 升级策略**。完成本章后，可继续 [第十五章 · NFT](../15_nft_kiosk/00-index.md) 或 [第十六章 · 客户端](../16_client/00-index.md) 做端到端练习。
